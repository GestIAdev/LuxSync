# WAVE 2122.3 — TEMPORAL UPSAMPLING: THE NYQUIST FIX

## ⚡ EXECUTIVE SUMMARY

WAVE 2122.2 eliminó el artefacto AGC (~95 BPM) pero reveló un problema más profundo: **resolución temporal insuficiente**. Con solo 14 lags enteros para cubrir 70-190 BPM, el medio-período a L7 (~185 BPM) dominaba sobre el beat real a L10 (~126 BPM). Se implementó **upsampling 4× del buffer de energía** antes de la autocorrelación, expandiendo la grilla de lags de [6,19] a [24,76] = 53 puntos. Resolución por bin: ~14 BPM/lag → ~3 BPM/lag.

**25/25 tests passed. Build OK. Performance: 69ms para 60s de audio (budget: 200ms).**

## 🔍 ROOT CAUSE: RESOLUCIÓN TEMPORAL (NYQUIST DE LAGS)

### Datos de producción WAVE 2122.2 (debugBPM.md)

```
AUTOCORR RAW (13 scans on ~126 BPM Brejcha):
  L7 (185bpm) = 0.19-0.62  ← DOMINANT (10/13 scans)
  L10(129bpm) = -0.22-0.21 ← REAL BEAT (barely visible)
  L14(92bpm)  = GONE        ← AGC artifact eliminated ✓
```

### El Problema Matemático

```
Frame duration: 46.4ms (2048 samples @ 44100 Hz)
Lag range [6,19] = 14 integer lags for 70-190 BPM

126 BPM = 476ms period = 10.26 frames/beat
  → Quantized to L10 = 464ms = 129 BPM
  → Each lag covers ~14 BPM of range
  → L7 = 325ms = 185 BPM (half-period modulation)

L7 sees the half-period energy modulation MORE CLEARLY
because it's a sharper feature on the coarse lag grid.
Parabolic interpolation can't rescue a peak that's
INVISIBLE because the grid is too coarse.
```

**Analogía**: Medir temperatura con un termómetro de 9°C de resolución.

## 🔧 THE FIX: 4× TEMPORAL UPSAMPLING

### Concepto

Antes de computar autocorrelación, interpolar linealmente el buffer de energía por factor 4:

```
Antes:  130 muestras, 46.4ms/muestra, lags [6,19]  = 14 puntos
Después: 520 muestras, 11.6ms/virtual,  lags [24,76] = 53 puntos

126 BPM: lag 10.26 (±7 BPM/bin) → lag 41.04 (±1.7 BPM/bin)
```

### Validez Matemática

La interpolación lineal es válida porque la señal de energía tiene bandwidth ~2 Hz (ritmo del beat), muy por debajo del Nyquist original (~10.8 Hz a 46.4ms/frame). No se inventa información — se revela estructura que existía pero estaba sub-muestreada en el dominio de lags.

### Cambios en el Código

**`GodEarBPMTracker.ts` (112 inserciones, 63 eliminaciones)**:

1. **Nueva constante**: `UPSAMPLE_FACTOR = 4`
2. **Nuevo campo**: `virtualFrameDurationMs = frameDurationMs / UPSAMPLE_FACTOR`
3. **Lag range recalculado**: Usa `virtualFrameDurationMs` para min/max lag
4. **`HARMONIC_LAG_TOLERANCE` 2→3**: Porque los lags son 4× más granulares
5. **Nuevo método `upsample()`**: Interpolación lineal de Float32Array
6. **`runAutocorrelationScan()` modificado**: Upsamplea antes de correlacionar, opera en espacio upsampled, convierte lags a BPM con `virtualFrameDurationMs`
7. **Diagnóstico actualizado**: `[🥁 AUTOCORR RAW 4×]` con `virtualDur=`, log cada 4 lags

### Costo Computacional

```
520 muestras × 53 lags = 27,560 multiply-adds por scan
Cada scan: cada 186ms (4 frames × 46.4ms)
CPU medido: 69ms para 60 segundos completos = NEGLIGIBLE
```
