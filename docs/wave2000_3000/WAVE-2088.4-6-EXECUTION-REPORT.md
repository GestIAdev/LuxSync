## 🔧 WAVE 2088.5 — DISCRETE PATTERNS → SMOOTH MOTION

### Objective
Make discrete patterns (square, botstep, chase_position) generate **continuous interpolated motion** instead of step-function jumps. Movers should never be "stuck" at a position — they should be transitioning toward the next one.

### Implementation

#### 1. **Square Pattern** — 4 Corners with Edge-to-Edge Interpolation
**File**: `src/engine/movement/VibeMovementManager.ts` (lines 239-265)

**Before**:
```typescript
square: (phase, audio) => {
  const normalizedPhase = (phase / (Math.PI * 2)) * 4
  const quadrant = Math.floor(normalizedPhase) % 4
  const corners = [
    { x: 1, y: 1 },
    { x: 1, y: -1 },
    { x: -1, y: -1 },
    { x: -1, y: 1 },
  ]
  return corners[quadrant]  // ← STEP FUNCTION: Same value for entire quadrant phase
}
```

**After**:
```typescript
square: (phase, audio) => {
  const corners = [
    { x: 1, y: 1 },
    { x: 1, y: -1 },
    { x: -1, y: -1 },
    { x: -1, y: 1 },
  ]
  const normalizedPhase = (phase / (Math.PI * 2)) * 4
  const currentCorner = Math.floor(normalizedPhase) % 4
  const nextCorner = (currentCorner + 1) % 4
  const t = normalizedPhase - Math.floor(normalizedPhase)  // 0..1 within segment
  
  // Hermite smoothstep: f(t) = 3t² - 2t³
  const eased = t * t * (3 - 2 * t)
  
  const from = corners[currentCorner]
  const to = corners[nextCorner]
  return {
    x: from.x + (to.x - from.x) * eased,
    y: from.y + (to.y - from.y) * eased,
  }
}
```

**Why Hermite?**
- Smooth acceleration/deceleration at segment boundaries
- f(0)=0, f(1)=1, f'(0)=0, f'(1)=0 → seamless motion through 4 corners
- Perceptually: mover slows at corners, speeds between them

```typescript
private translateToDriverPhysicsProfile(rawProfile: any): DriverPhysicsProfile | null {
  if (!rawProfile) return null
  
  // ═══════════════════════════════════════════════════════════════════════
  // SOURCE A: ShowFileV2.PhysicsProfile (fixture.physics from Forge v2)
  // Has: motorType='servo-pro'|'stepper-quality'|'stepper-cheap'|'unknown'
  //      maxAcceleration, maxVelocity (already in DMX/s)
  // ═══════════════════════════════════════════════════════════════════════
  if (rawProfile.motorType && rawProfile.maxVelocity !== undefined && rawProfile.orientation !== undefined) {
    const motorTypeMap: Record<string, 'stepper' | 'servo' | 'unknown'> = {
      'servo-pro': 'servo',
      'stepper-quality': 'stepper',
      'stepper-cheap': 'stepper',
      'unknown': 'unknown',
    }
    const qualityMap: Record<string, 'budget' | 'mid' | 'pro'> = {
      'servo-pro': 'pro',
      'stepper-quality': 'mid',
      'stepper-cheap': 'budget',
      'unknown': 'budget',
    }
    
    // Robe Robin reference: ~300°/s = ~142 DMX/s
    // speedFactor = 0.5 means this mover is 50% as fast as Robe Robin
    const REF_SPEED_DMX = 142
    const speedFactor = Math.min(1.0, rawProfile.maxVelocity / REF_SPEED_DMX)
    
    return {
      motorType: motorTypeMap[rawProfile.motorType] ?? 'unknown',
      maxAcceleration: rawProfile.maxAcceleration,
      maxVelocity: rawProfile.maxVelocity,
      panSpeedFactor: speedFactor,
      tiltSpeedFactor: speedFactor * 0.7,  // Tilt is always slower (gravity)
      qualityTier: qualityMap[rawProfile.motorType] ?? 'budget',
    }
  }
  
  // ═══════════════════════════════════════════════════════════════════════
  // SOURCE B: FixtureProfiles.movement (from getProfile/getProfileByModel)
  // Has: movement.type='stepper', movement.maxPanSpeed=180 (DEGREES/s!)
  // ═══════════════════════════════════════════════════════════════════════
  if (rawProfile.movement && rawProfile.movement.maxPanSpeed !== undefined) {
    const mov = rawProfile.movement
    
    // Convert degrees/s to speedFactor relative to Robe Robin (300°/s)
    const REF_SPEED_DEG = 300
    const panFactor = Math.min(1.0, mov.maxPanSpeed / REF_SPEED_DEG)
    const tiltFactor = Math.min(1.0, (mov.maxTiltSpeed ?? mov.maxPanSpeed * 0.7) / (REF_SPEED_DEG * 0.7))
    
    // Infer qualityTier from speed
    let qualityTier: 'budget' | 'mid' | 'pro' = 'budget'
    if (mov.maxPanSpeed >= 250) qualityTier = 'pro'
    else if (mov.maxPanSpeed >= 160) qualityTier = 'mid'
    
    const motorTypeMap: Record<string, 'stepper' | 'servo' | 'unknown'> = {
      'stepper': 'stepper',
      'servo': 'servo',
      'galvo': 'servo',
    }
    
    return {
      motorType: motorTypeMap[mov.type] ?? 'stepper',
      maxAcceleration: undefined,  // FixtureProfiles doesn't have accel
      maxVelocity: undefined,
      panSpeedFactor: panFactor,
      tiltSpeedFactor: tiltFactor,
      qualityTier,
    }
  }
  
  // ═══════════════════════════════════════════════════════════════════════
  // SOURCE C: Already a DriverPhysicsProfile (passthrough)
  // ═══════════════════════════════════════════════════════════════════════
  if (rawProfile.panSpeedFactor !== undefined || rawProfile.qualityTier !== undefined) {
    return rawProfile as DriverPhysicsProfile
  }
  
  return null
}
```

  maxAcceleration: number
  maxVelocity: number
  speedFactorPan: number
  speedFactorTilt: number

  const profile = config.physicsProfile  // NOW POPULATED!
  
  // Hierarchy of safety (most restrictive wins):
  // 1. SAFETY_CAP (hardcoded, never violated)
  // 2. Vibe Request (what the music genre asks for)

  if (profile) {

  const speedFactorPan = profile?.panSpeedFactor ?? 1.0
  const speedFactorTilt = profile?.tiltSpeedFactor ?? 1.0
  
  return { maxAcceleration, maxVelocity: effectiveMaxVel, speedFactorPan, speedFactorTilt }
}

## 📝 TECHNICAL NOTES

### Why Hermite Smoothstep?
```
f(t) = 3t² - 2t³

Properties:
- f(0) = 0, f(1) = 1 (maps perfectly to [0,1])
- f'(0) = 0, f'(1) = 0 (zero velocity at boundaries → smooth joints)
- Symmetric around t=0.5 (natural rhythm for movement)
- Faster in middle, slower at edges (acceleration/deceleration profile)

Alternative considered: Linear interpolation
- Would work mathematically
- But visually less smooth, jerky at corners
- Hermite is the industry standard for animation
```

### Why 135°/s for Budget Movers?
```
Empirical data from cheap Chinese movers:
- Stepper motor specs: ~1.8°/step × 100 steps/s = 180°/s max
- But with gearbox reduction + mechanical play: ~50-70% effective
- Realistic sustained speed: 100-135°/s
- We picked 135°/s as "good day" performance
- This gives speedFactor = 135/300 = 0.45
```

### Why speedFactor Instead of Absolute Velocity?
```
Could have stored absolute maxVelocity in the profile:
- Pro: Direct, no reference frame needed
- Con: Breaks abstraction — Physics Driver shouldn't know about degrees

Using speedFactor:
- Pro: Dimensionless, scales with preset expectations
- Con: Requires reference frame (Robe Robin = 1.0)
- Better: Preset says "140 DMX/s for pro" → budget gets 0.45×140 = 63 DMX/s
```
